ESP32-S3 LVGL IoT Dashboard
===========================

> ESP32-S3 based smart HMI panel with LVGL UI, Wi-Fi networking, weather display, SD card media, and custom splash animation.

## Overview

This project implements a small **IoT-style dashboard** on an **ESP32-S3** development board.  
It uses **LVGL** as the GUI framework to build multiple screens (splash screen, password/main screen, music UI, etc.), integrates:

- Wi-Fi networking
- Weather information display
- File systems (SD card via FATFS, SPIFFS)
- Simple multimedia (MP3 audio playback, image/animation display)

The focus is on **embedded engineering and system integration**: making all peripherals and software components work together smoothly on a resource-constrained MCU.

## Features

- **LVGL-based GUI**
  - Multiple screens generated by NXP Gui Guider, with manual customization.
  - Custom **splash screen** using `lv_animimg` (full-screen animation played once on startup, then auto-switch to main UI).
  - Touch input handling and basic UI navigation.

- **Wi-Fi & Networking**
  - Initializes Wi-Fi protocol stack early in `app_main` to reduce first-time entry latency.
  - Connects to an access point in station mode (implementation in custom modules, e.g. `custom_init` / `app_wifi_init`).
  - Designed to fetch online data (e.g. weather) via HTTP/REST APIs.

- **Weather Display**
  - UI elements for showing **current city, temperature, and weather status**.
  - Weather data is obtained via network requests and displayed on the LVGL screen.

- **File Systems**
  - **SD card** mounted at `/sdcard` using `esp_vfs_fat_sdmmc_mount`.
  - **SPIFFS** mounted at `/spiffs` for internal resources.
  - Custom LVGL filesystem driver `sd_lvgl_fs`:
    - Maps LVGL drive letter **`S:`** to VFS path **`/sdcard`**.
    - LVGL widgets can load images/resources directly from the SD card (e.g. PNG, GIF, etc.).
  - FATFS Long File Name (LFN) support enabled to handle long filenames such as `frame_0001.jpg`.

- **Multimedia**
  - MP3 playback from SD card using the board’s audio codec and `bsp_codec_init`.
  - SD-stored images and animations integrated into LVGL UI.
  - Experiments with GIF, multi-JPEG (MJPEG-style), `lv_animimg` with embedded frames, and performance tuning.

- **Performance & Resource Optimization**
  - Tuned LVGL display buffer size, buffering mode, DMA usage, and PSRAM/SRAM placement.
  - Reduced flash usage by:
    - Limiting the number of embedded animation frames.
    - Disabling unused LVGL image decoders (PNG/JPG) once not needed.
  - Addressed issues such as:
    - App partition overflow due to large `.rodata` (embedded images).
    - Screen tearing caused by too-high frame rate for full-screen animations.

## Hardware Requirements

- ESP32-S3 development board (e.g. LCSC “实战派 S3” or similar) with:
  - SPI or RGB TFT LCD panel (e.g. 320x240)
  - Capacitive or resistive touch controller (connected via I2C)
  - On-board audio codec (for MP3 playback)
  - SD card slot (SDMMC 1-bit mode in this project)
- microSD card (formatted as FAT/FAT32)
- USB cable for flashing and serial monitor

## Software Stack

- **ESP-IDF** (v5.x recommended)
- **LVGL** (via managed component `lvgl__lvgl`, configured in `lv_conf.h` and `sdkconfig`)
- **NXP Gui Guider** (used to generate initial LVGL screens and layout)
- **CMake** build system (ESP-IDF default)

## Project Structure (simplified)

- `main/`
  - `main.c`  
    - Entry point: initializes NVS, Wi-Fi stack (`app_wifi_init`), I2C, audio codec, SD card (`sd_card_mount`), SPIFFS (`spiffs_mount`), LVGL port (`bsp_lvgl_start`), LVGL filesystem drivers (`lv_fs_posix_init`, `lv_fs_sdcard_init`), and UI (`setup_ui`, `custom_init`).
  - `esp32_s3_szp.c`  
    - Board support package: LCD, touch, audio codec, LVGL port configuration (display buffer, DMA, PSRAM/SRAM).
  - `sd_lvgl_fs.c/.h`  
    - Custom LVGL filesystem driver that maps `S:` → `/sdcard`, implemented using standard C `fopen`/`fread` on top of ESP-IDF VFS.
  - `custom.c/.h`  
    - Custom application logic: Wi-Fi, music player UI integration, weather display, etc.
  - `app_ui*.c`  
    - Additional UI logic for specific screens (e.g. music control page).
  - `generated/`  
    - Auto-generated LVGL UI code from NXP Gui Guider:
      - `gui_guider.c/.h` – top-level UI structure and `setup_ui`.
      - `setup_scr_*.c` – per-screen setup (widgets, layout, styles).
      - `events_init.c` – LVGL event callbacks.
      - `widgets_init.c/.h` – widget and image/animation resources.

## Build & Flash

1. **Install ESP-IDF**

   Follow the official ESP-IDF documentation to install the toolchain and set up the environment:

   ```bash
   # Example (Linux/macOS)
   git clone --recursive https://github.com/espressif/esp-idf.git
   cd esp-idf
   ./install.sh
   . ./export.sh
   ```

2. **Configure the Project**

   In the project root:

   ```bash
   idf.py menuconfig
   ```

   Check or adjust:

   - Serial port, flash size, and partition table.
   - Wi-Fi SSID/password (if stored in `sdkconfig` or custom config).
   - FATFS LFN options:
     - `CONFIG_FATFS_LFN_HEAP=y`
     - `CONFIG_FATFS_MAX_LFN=255`
   - LVGL options:
     - Enable/disable image decoders (GIF/PNG/JPG) as needed.
     - FS options (e.g. `LV_USE_FS_POSIX`) if used.

3. **Build**

   ```bash
   idf.py build
   ```

4. **Flash & Monitor**

   ```bash
   idf.py -p COMx flash monitor
   # Replace COMx with your serial port on Windows
   ```

## SD Card & Resources

- SD card is mounted at `/sdcard` in ESP-IDF.
- LVGL accesses the same SD card via logical drive `S:` (e.g. `S:/gif/xxx.gif`, `S:/img/xxx.bin`).
- Typical directory layout on SD card:

  ```text
  /gif        - GIF animation files (if used)
  /music      - MP3 or other audio files
  /images     - PNG/JPG/BIN image resources
  ```

Make sure the SD card is formatted as FAT/FAT32 and inserted before boot.

## Known Issues & Notes

- Full-screen animations at high frame rate (e.g. > 15 fps at 320x240) can cause visible tearing due to SPI/LCD bandwidth limits.  
  This project chooses a **lower frame rate or reduced resolution** to get smoother perceived motion.
- Embedding too many 320x240 frames as C arrays will quickly bloat `.rodata` and can overflow the app partition.  
  Consider:
  - Reducing number of frames.
  - Using external resources on SD card.
  - Using LVGL’s binary image format (`.bin`) for faster loading.
